Single Static Assignment (SSA) Revisited
• SSA is an intermediate representation where each
variable is assigned exactly once. This simplifies analysis
because a variable’s definition is clearly distinct from its
uses.
• In SSA, each variable is given a unique version
whenever it is redefined.
- For example, instead of just x, we may have x1, x2,
x3, etc., for each definition of x.
• SSA simplifies liveness analysis and other optimizations
because the flow of data is more explicit (each variable is
defined once and only once).

Why Convert Stack-Based LLVM IR to
Virtual Register SSA?
• What We Have (Stack-Based IR)
- Variables stored in memory (alloca, load, store)
- Values repeatedly written to and read from stack slots
- Not in true SSA form
- Hard to reason about data dependencies
• What We Want (Virtual Register SSA)
- Each variable becomes a virtual register
- Each value defined exactly once (SSA)
- No unnecessary memory traffic

φ-Nodes
• When converting from stack-based IR to SSA:
- Each variable must be assigned exactly once
- Different control-flow paths may assign different
values
- At merge points, we must reconcile those values
• Without φ:
- We cannot represent “x depends on which path we
took”
- φ-nodes solve this problem.

Aside: Phi Instructions
• A LLVM phi instruction is a special instruction
used in SSA form to merge different versions of
a variable at control flow joins (e.g., at the
beginning of basic blocks where multiple paths
converge).
- LLVM phi instructions appear at the entry of
a basic block, where there are multiple
predecessors, and it merges the incoming
values from those predecessors.
• Specifically, they merge different definitions
of a variable that come from multiple
control flow paths (e.g., branches or loops).
• Phi instructions are necessary in SSA form to
maintain the rule that every variable is assigned
exactly once.

func foo(x int) int {
var sum int;
if (x > 3)
{
sum = x + 2;
} else {
sum = x + 1;
}
return sum;
}

Sum will either be x+2 or x+1 based on the control-flow of the value of x.

Phi Instruction
%result = phi <type> [<value1>, <block1>], [<value2>, <block2>], ...

• <result>: This is the result variable that will hold the merged
value.
• <type>: The type of the value being merged (e.g., i64 for 64bit integers).
• <value1> and <value2>: These are the values coming from
different predecessor blocks. Each value corresponds to a
definition from a predecessor block.
• <block1> and <block2>: These are the predecessor blocks
where each value originates. The control flow determines
which value the phi node will pick based on the path taken.

MPCS 51300 - Compilers

9

Phi Instructions
• The number of operands for a phi node depends on how
many predecessor blocks a basic block has. Each
predecessor contributes an operand (a potential value) for
the phi instruction.
• If a basic block has N predecessors, the phi instruction will
have N operands (each representing a possible value
coming from a predecessor).

Dataflow Analysis
• Dataflow analysis is a technique for computing
information that flows through a program’s control-flow
graph (CFG).
• It answers questions like:
- What definitions reach this point?
- What variables are live?
- What is the current value of a variable?
• It works by:
- Associating information with each block
- Propagating that information along CFG edges
- Iterating until the information stabilizes
MPCS 51300 - Compilers

13

Forward vs Backward Dataflow
• Forward analysis
- Information flows in direction of control flow
- Example: reaching definitions, constant propagation
- Our SSA construction is forward
• Backward analysis
- Information flows opposite control flow
- Example: liveness analysis(we’ll talk about this when
we discuss register allocation and optimizations)

Defs and Uses
• Def (Definition):
- A "definition" of a variable occurs when a value is assigned to
that variable.

store i64 %v, i64* %x ; de nes x
• Use (Definition): %t = add i64 %a, %b ; de nes %t
- A “use” occurs when a variable is read, but not assigned a new
value.

%t = load i64, i64* %x ; uses x
%u = add i64 %a, %b
; uses %a and %b

• A variable must be defined before it can be used.
• In SSA construction:
- We track definitions of stack variables
- We replace uses (loads) with their current SSA value

Aside: What Is a Fixed-Point Algorithm?
• A fixed-point algorithm:
- Repeatedly updates information
- Stops when no changes occur
• Formally:
- A fixed point is reached when applying the algorithm
again produces no further changes.
• Why needed?
- SSA Construction: Loops create cyclic dependencies
- Information must circulate through cycles
- One pass is not enough

SSA via Dataflow (High-Level Idea)
• Let B be a Basic block, lets define IN and OUT maps:
- IN[B] = variable → SSA value at block entry
- OUT[B] = variable → SSA value at block exit
• Algorithm idea:
- Merge predecessor OUT maps to form IN
- Insert φ when predecessors disagree
- Process instructions to compute OUT
- Repeat until stable

What Does “Stable” Mean?
• The algorithm stops when:
For all blocks B:
OUT[B] did not change this iteration

• That means:
- No new φ inserted
- No mappings updated
- Information fully propagated
- We have reached a fixed point.

Aside: Read Statement
• In Golite, the read statement (scan x) reads a single

global memory address) to store the value of the
stack variable.
- You then load the value from this global variable into
a new temporary register.
- Update the map so that the new temporary register
represents the current value of the stack variable.

Liveness Analysis

44

What is Liveness Analysis?
• A technique used in compiler optimizations to determine
which variables are live at each point in a program. A variable
is considered live
- If its value may be needed in the future (i.e., if it will be
used before it is overwritten).
- Generally speaking, it is live between its definition and its
last use.
• Liveness analysis helps identify opportunities for
optimization, such as register allocation or dead code
elimination.
- In this lecture, we will focus on register allocation and
come back to liveness analysis when talking about other
optimizations.
MPCS 51300 - Compilers

45

Live Ranges
• The live range represent the span of time a variable is
live in a program.
• The live range begins at the point where the variable is
first used and ends when it is last used or overwritten.
- If a variable is defined an instruction ix and last used
at an instruction iy then it’s live range is [ix, ij]

Critical Edge Problem
• In a control-flow graph (CFG), An edge A → B is a critical

- A has multiple successors
- B has multiple predecessors

• Why Do We Care?
- Because when removing φ-nodes, we must insert
copies on specific edges.
- If the edge is critical, we have no safe place to put the

Linear Scan Register Allocation
(Step 1: Compute Live Intervals)
Even though SSA guarantees each virtual register is defined only once, you still need to
determine where each value is "alive." You do this by:
• A) Make sure your SSA is in Out-of-SSA form:
- Insert Pseudo-LLVM Move Instruction ( e.g. mov %op_reg, %dest_reg). You
need to define this instruction.
- For each PHI operand, insert a pseudo LLVM move at the end of its
corresponding predecessor block. This move transfers the value from the
PHI’s operand’s assigned physical register to the physical register designated
for the Phi’s destination register, effectively converting the IR out of SSA form.
Make sure to consider the critical edge special case.
• After updating, the LLVM IR will no longer be valid LLVM IR!
• B) Linearizing the code:
- Assign each instruction a unique position (By performing a Reverse
Postorder over the CFG for each function and assigning each instruction a
unique position).
• C) Compute live ranges for each virtual register
- Live Range Start: The position where a virtual register is defined.
- Live Range End: The position of its last use across all blocks.

MPCS 51300 - Compilers

60

Linear Scan Register Allocation
(Step 2: Perform Linear Scan)
Once you have live intervals for each SSA register, you perform the
linear scan allocation:
• A) Sort Intervals:
- Arrange the live intervals in increasing order of their start points.
• B) Keep an active set, which is the set of virtual registers currently
assigned to a physical register. As you iterate through the intervals:
- Expire Old Intervals: Remove intervals from the active set that
have ended (i.e., their end point is before the current interval’s
start).
- Assign a Register:
‣ If a free register is available, assign it to the current interval.
‣ If not, decide which interval to spill (often the one with the
farthest end point) or split the current interval if beneficial.

Determining number of Registers
• How many physical registers do you have? It depends on the
target architecture.
- The amount you have available is all registers minus
the reserved registers for the architecture.
- ARM64 has at least 7 reserved registers: {x29, x30,
sp, SPILL_REG1, SPILL_REG2, x16, x17, x18}, where
SPILL_REG1 and SPILL_RE21 is discussed below.
• Spilling code involves going back to the stack-based
representation where you need loads/stores directly to the
stack.
- You need at least two physical register to be dedicated to
spilling registers. For ARM64, I recommend you reserve
SPILL_REG{1,2} = x9,x10.
