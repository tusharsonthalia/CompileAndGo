ARM System Architecture Overview
• ARM stands for Advanced RISC Machines
- Originated in England in 1984, where it was mostly used in British educational
systems
- Late 80s and early 90s, ARM became the first commercial RISC processor and
is now being used in many devices for its low power-energy usage and simple
instruction set.
• Why use a RISC based architecture?
- First, RISC stands for "Reduced Instruction Set Computer", contrasted with a
CISC machine (Complex Instruction Set Computer, Intel).
- RISC Advantages
‣ Smaller die sizes
‣ Shorter time to develop since it has a simple instruction set.
‣ If optimized correctly, it can be more energy efficient.
- RISC Disadvantages
‣ Generally less code density than CISC
‣ Cannot execute x86 code (i.e., intel) without some type of conversion.
Apple has this conversion, which is known as Rosetta 2 for its new ARM
Macs, in order to execute Intel-compiled programs.

ARM Program Structure
• An ARM assembly file contains a mixture of assembly
directives, instructions, and labels:
• Assembler directives are directions to the assembler
to take some action or change a setting
• They always begin with a period (“.”)
• .text states the following lines will go inside the
code segment of the virtual address space.
• .arch armv8-a - states that this assembly file
works on ARMV8-a architectures
• The remaining code in an ARM program are ARM
instructions and labels.

ARM Program Structure: Functions
• The remainder of the assembly file after .text will be functions along with their
assembly code. Each must have the following structure:
.type myFunc, %function
.global myFunc
.p2align
2 // All functions must be 16-bit aligned (i.e. 2 byte aligned)
myFunc: // the start of the myFunc function
// myFunc function code goes here after the label
.size myFunc, (. - myFunc) //end of the function always ends with .size

• .type name, type_description - directive sets the type of a symbol name to be
either a function or an object. %function is the type_description for functions.
• .global name - directive makes the function accessible anywhere in the file or
outside the file for external files.
• .size name, expression - directive is used to set the size associated with a
function. This information helps the linker to exclude unneeded code and/or data
when creating an executable file, and helps the debugger to keep track of where it
is in the program.
• (. - myFunc) - means subtract the location of the start of the function label
(myFunc) to the current location of .size. This directive calculates how many
bytes there are between the label and .size and is useful for a debugger/
linker.

Procedure Call Standard ARM64
• All return values for functions that return a value are stored in register 0.
Register
x0

Size
64 bits

• Registers R0-R15 are known as caller-saved temporary registers
- From ARM: “If the caller requires the values in any of these registers to be
preserved across a call to another function, the caller must save the affected
registers in its own stack frame. They can be modified by the called subroutine
without the need to save and restore them before returning to the caller.”
• Make sure to save these to the stack BEFORE calling the function.
• Registers R19-R29 are know as callee-saved registers
- From ARM: “These registers are saved in the callee frame. They can be
modified by the called subroutine as long as they are saved and restored before
returning.”
- This means that if the callee function is required to use registers pass 19 that
before you start executing the callee function you must first save the current
values of these registers in the callee frame and before exiting you must restore
the values
- Do not use SP or R30 ( Link Register).

Function Prologue & Epilogue
• When entering and exiting a function you must push or
pop off the frame pointer and link register.
• Prologue - must be performed at the start of the code
within the callee function
sub sp, sp, 16
// Decrement SP to allocate space for previous frame’s LR and FP
stp x29, x30, [sp]
// [sp] = x29; [sp + 8] = x30
mov x29, sp
// FP points to frame record
// You’ll need to update sp again to make space for the entire incoming frame or you can
// from the beginning.

• Epilogue
ldp x29, x30, [sp]
// x29 = [sp]; x30 = [sp + 8]
add sp, sp, 16
// “deallocate” the space for the FP and LR
ret
// You will need to make sure before this code or after that you increment sp to remove the
// space for the callee frame.

Malloc runtime call
• In Golite, you will need to allocate structs values on the
heap. Therefore you will need to use malloc. Before
calling malloc you must store the number of bytes you
wish to allocate into r0.
• The address to the allocated memory will be stored
back into r0 when the function returns.
type Point struct {
x int;
y int;
};
func main() {
var p *Point;
p = new Point;
}

bl malloc
// Now x8 has the address to the allocated memory for
the Point since malloc stores the return value in r0.
mov x8, x0

Printing call
• Similar to Printf in Go, you need to specify the formatter string and the variables to print.
These are one or more separate arguments.
- The first argument must be the format string and the address must be placed in r0.
Similar to LLVM IR, each format string should have a label associated with it. You
need to specify the .asciz and .size directors that represents the string in ascii
format and the size plus space for the null character.
- Similar to global variables always use adrp and add to get the address of the
format string.
- Key things: x0 must always be the address of the format string (e.g. .FMTSTR1),
and the registers (and possibly stack) hold the value of the expressions to print.
Define the following labels as
the very last labels in the file.
.FMTSTR1:
.asciz
"x=%ld"
.size
.PRINT, 6

// I’m not showing all the code here just what’s
needed for printing purposes
adrp
x10, .FMTSTR1
add
x10, x10, :lo12:.FMTSTR1
mov x0, x10
mov x1, x4 // Assume x is assigned to x4
bl printf

x = 4;
printf(“x=%d”,x);

Reading in an Integer
• Similar to Scanf in Go, you need to specify the address of the variable to place the read in value
along with the format string for reading in an int64. These are two separate arguments.
- The first argument must be the format string and the address must be placed in r0. Use the
following code to always create the format string. This will not change for any program.
- Similar to global variables always use adrp and add to get the address of the format string.
- Scanf will store the value read in into the address of x1. You will need to perform a
load operation to place that value into a register before using it!
- Key things: r0 must always be the address of the either .READ. r1 holds the address of the
variable to place the read in value.

Register Allocation
• LLVM uses an infinite number of registers when generating code
but when translating too assembly we only have a finite number
of registers to use.
• Thus, we need to have use a register allocation scheme
• A process that assigned variables and temporaries to
registers and manages the transfer of data in and out of
registers.
• For Golite compiler, we are first going to use a stack allocation
approach.
• This approach is NEVER used in production compilers;
however, it’s a simple approach to understanding the
generation of IR code to assembly.
• A better approach is to use a graph-coloring register
allocation.

Stack Based Allocator
• The naive register allocation approach is as follows:
• Allocate enough space on the stack for all arguments,
locals, and temp variables.
• You must still adhere the ARM 64 Procedure Call
standard (e.g., x0-x7 will always hold the first
arguments).
• There will be a large number of load/store calls based
on always accessing and updating the stack.
• Note: This scheme will only work on small programs
since this scheme could potentially cause a stack
overflow.

Analysis of Naive Register Allocation
• Disadvantage: Horribly inefficient
• Generates way too many unnecessary load and store instructions
• Wastes space on temporaries that can be stored in unused
registers
• Memory loads and stores are a hinderance to performance to this
version will result in slower programs
• Never used in production compilers
• Advantage: Simplicity
• Can easily translate each IR instruction directly to assembly.
• Never need to worry about running out of registers, function calls
or special-purpose registers.
• From a pedagogy perspective, it’s great for quickly implementing
code generation for a prototype language such as Golite.
• Recommend using a graph-coloring register allocation.

LLVM to ARM Assembly
• Similar to AST to LLVM, one way to implement the translation
from LLVM to ARM is to define a method for each LLVM
instruction that will translate its LLVM representation to assembly.
For example, here’s how the Add ILOC instruction could look like:
func (instr *Add) TranslateToAssembly(tables * st.SymbolTables)
[]codegen.Instruction {
}

• Overall, you should define a codegen package that contains a
function that translates your function fragments into a code
generation function
•

func TranslateToAssembly([]*BasicBlock, tables *
st.SymbolTables) ... ... {
}
